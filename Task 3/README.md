# Переход на Event-Driven архитектуру

## Список проблем текущей архитектуры

| № | Проблема | Описание |
|---|----------|----------|
| **P1** | **Синхронные цепочки вызовов** | `core-app` → `ins-product-aggregator` → 5 страховых. Любая задержка или ошибка у одной страховой блокирует весь запрос. |
| **P2** | **Устаревшие локальные реплики** | `core-app` обновляет данные раз в 15 мин, `ins-comp-settlement` — раз в сутки. Между обновлениями сервисы работают с неактуальными данными. |
| **P3** | **Каскадные ошибки** | Ошибка в `ins-product-aggregator` или любой страховой приводит к ошибке в `core-app` и `ins-comp-settlement`. |
| **P4** | **Высокая нагрузка в моменты обновления** | Все экземпляры `core-app` одновременно запрашивают агрегатор каждые 15 мин → пики нагрузки. |
| **P5** | **Отсутствие retry и circuit breaker** | При временных сбоях страховых API нет механизмов повторных попыток или "отсечки" проблемного сервиса. |

## Риски при подключении ещё 5 страховых компаний

| № | Риск | Вероятность | Влияние | Описание |
|---|------|-------------|---------|----------|
| **R1** | **Рост времени ответа** | Высокая | Критическое | Время синхронного вызова вырастет с 5×T до 10×T. При T=1 сек → 10 сек ответа. |
| **R2** | **Увеличение числа ошибок** | Высокая | Высокое | Вероятность ошибки хотя бы у одной из 10 страховых: 1 - (1-p)^10. При p=1% → 9.6% ошибок. |
| **R3** | **Рост нагрузки на агрегатор** | Высокая | Среднее | 10 страховых → в 2 раза больше исходящих запросов и обработки. |
| **R4** | **Устаревание данных** | Средняя | Среднее | При 10 страховых данные меняются чаще, а частота обновления реплик остаётся прежней. |
| **R5** | **Сложность отладки** | Средняя | Низкое | При 10 источниках данных сложнее определить, какая страховая вернула ошибку или некорректные данные. |

## Предлагаемые решения

| Проблема/Риск | Решение | Паттерн |
|--------------|---------|---------|
| P1, R1, R2 | Переход на асинхронное обновление данных через событийную шину | Event Streaming |
| P2, R4 | Реальное время (near-real-time) обновления реплик через CDC | Change Data Capture |
| P3, R5 | Circuit Breaker + Retry + Dead Letter Queue | Resilience Patterns |
| P4 | Jittered scheduled updates + кэширование | Thundering Herd Prevention |
| P5, R3 | Внедрение retry с exponential backoff | Retry Pattern |

Паттерн Transactional Outbox: 

**Обоснование:**
- Обеспечивает атомарность между записью в БД и отправкой события
- Гарантирует доставку событий (at-least-once)
- Позволяет восстановить события после сбоев
