# Технологическая архитектура InsureTech (геораспределённая)

## 1. Стратегия масштабирования и отказоустойчивости

### Вертикальное vs горизонтальное масштабирование

| Параметр | Вертикальное | Горизонтальное (выбрано) |
|----------|--------------|--------------------------|
| **Отказоустойчивость** | Низкая (одна точка отказа) | Высокая (поды/узлы) |
| **RTO 45 мин** | Недостижим (перезапуск ВМ долгий) | Автоматическое восстановление |
| **Рост регионов РФ** | Ограничен лимитами одной ВМ | Добавляем узлы/зоны |
| **Цена** | Дешевле до порога | Дороже, но окупается SLA |

**Решение**: только горизонтальное.  
Вертикальное — только для control plane (etcd, API server) до 3–5 узлов.

### Дополнительные зоны безопасности (Availability Zones)

- Используем **3 AZ** в каждом федеральном округе (минимум 2 AZ для RTO).
- Зоны изолированы по питанию/сети — обеспечивают 99.99% внутри региона.

## 2. Деплой Kubernetes: независимые кластеры в каждой площадке

**Выбрано**: отдельные кластеры K8s в каждом крупном регионе РФ (Москва, СПб, Екатеринбург, Новосибирск, Владивосток).

## 3. Балансировка нагрузки (геораспределённая)

### Балансировка на уровне сервисов

- Внутри кластера: Kubernetes Service (ClusterIP) + kube-proxy (iptables/IPVS).
- Для публичных API: LoadBalancer → Ingress Controller

## 4. Фейловер-стратегия (RTO 45 мин, RPO 15 мин)

**Выбрано**: Active-Passive с региональным переключением + глобальный GeoDNS.

### Схема фейловера (на уровне клиент-приложение)

Шардирование не требуется до 2 TB данных.
При росте — партиционирование по client_id (хеш-шарды).

Фейловер-стратегия: Active-Passive с GeoDNS.
При недоступности region-health >30 сек → переключение на другой регион.

## 5. Конфигурация базы данных (PostgreSQL)

### Репликация и RPO/RTO

- **Внутри региона**: синхронная репликация (synchronous_commit = on) → RPO = 0.
- **Между регионами**: асинхронная логическая репликация (публикация/подписка) с WAL retention 30 мин.  
  RPO = 15 мин (при падении лидера в Москве теряется не более 15 мин данных).
- **Бэкапы**: pg_basebackup + WAL архив в S3-совместимое хранилище каждые 6 часов.  
  Дополнительно: point-in-time recovery (PITR) до любой минуты за последние 7 дней.
